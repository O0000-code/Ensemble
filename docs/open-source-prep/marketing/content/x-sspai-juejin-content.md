## 三、文章开头（500字，引人入胜）

> **用 Ensemble 告别 Claude Code 配置混乱：我的 AI 编程效率提升之道**

作为一名效率工具爱好者，我对 Claude Code 的第一印象是「惊艳」。它能理解我的代码意图，帮我重构复杂逻辑，甚至能直接操作文件系统。当我发现 MCP（Model Context Protocol）能让它连接数据库、调用 API 时，我几乎是迫不及待地安装了十几个 MCP Server。

然而惊艳过后，困扰随之而来。

第一个问题是**上下文空间告急**。Claude Code 的上下文窗口是有限的，而我安装的每一个 MCP Server、每一个 Skill，无论这次对话是否需要，都会被加载到上下文中。就像一个塞满工具的工具箱，每次出门都要全部背上——即使你只是去拧一颗螺丝。我开始发现 Claude 的回复变得「健忘」，长对话中它会丢失前文信息，因为真正有用的对话内容被工具描述挤占了空间。

第二个问题是**配置管理的混乱**。Skills 配置在一个目录，MCP 配置在另一个地方，CLAUDE.md 又分散在各个项目里。每次换项目都要想一想：「这个项目需要哪些工具？上次我是怎么配的？」手动编辑 JSON 文件的体验让我这个追求效率的人感到窒息。

第三个问题是**场景切换的低效**。我的前端项目需要 Figma MCP 和 React 相关 Skills；数据分析项目需要 PostgreSQL MCP 和 Python Skills；而写文档时我只想要一个干净清爽的环境。每次切换都是一场配置文件的修改大战。

于是我问自己：**能不能有一个工具，让我按场景管理这些配置，需要什么启用什么，一键切换，干净利落？**

答案就是 Ensemble。

这是一款运行在 macOS 上的桌面应用，它用「三位一体」的方式管理 Claude Code 的所有配置——Skills、MCP Servers、CLAUDE.md。更重要的是，它引入了「Scenes」（场景）的概念，让你可以像切换 Focus Mode 一样切换 Claude Code 的工具集。

接下来，让我详细介绍这款工具是如何解决我的配置焦虑的。

## 四、核心卖点段落（详细介绍三位一体管理）

### 三位一体：重新定义 Claude Code 配置管理

如果把 Claude Code 比作一位万能的助手，那么 Skills、MCP Servers 和 CLAUDE.md 就是赋予它能力的三大支柱。Ensemble 的核心设计理念，就是将这三者纳入统一的管理体系。

#### Skills 管理：从混乱到清晰

Skills 是 Claude Code 的内置能力扩展。Ensemble 会自动扫描系统中已安装的所有 Skills，并以清晰的列表形式展示。你可以：

- **浏览和搜索**：快速找到需要的 Skill
- **分类和标签**：按用途给 Skills 打标签（前端、后端、数据等）
- **AI 智能分类**：Ensemble 内置的分类功能可以根据 Skill 描述自动建议分类
- **启用/禁用**：一键控制哪些 Skills 参与当前场景

不再需要手动编辑 skills.json，不再需要记忆每个 Skill 的文件路径。

#### MCP Servers 管理：可视化配置的魅力

MCP Servers 是 Claude Code 连接外部世界的桥梁——数据库、API、第三方服务都通过 MCP 接入。Ensemble 提供了完整的 MCP 管理功能：

- **服务器配置**：可视化编辑 MCP Server 的配置参数
- **工具发现**：自动列出每个 MCP Server 提供的所有工具
- **连接状态**：实时查看各服务器的运行状态
- **快速添加**：内置常用 MCP Server 模板，一键添加

告别在 JSON 配置文件中反复试错的痛苦。

#### CLAUDE.md 管理：指令的艺术

CLAUDE.md 是写给 Claude Code 的「使用说明书」，它定义了 Claude 应该如何理解和处理你的项目。Ensemble 让 CLAUDE.md 的管理变得系统化：

- **全局设置**：定义适用于所有项目的通用指令
- **项目分发**：将配置好的 CLAUDE.md 同步到指定项目
- **版本管理**：不同场景可以使用不同的 CLAUDE.md 模板
- **快速编辑**：内置编辑器，所见即所得

#### Scenes：场景的力量

这三大功能的真正威力，在于「Scenes」（场景）的整合。一个 Scene 就是一套完整的配置预设：

```
前端开发场景 = Skills(React, TypeScript) + MCP(Figma, Vercel) + CLAUDE.md(前端规范)
数据分析场景 = Skills(Python, Pandas) + MCP(PostgreSQL, BigQuery) + CLAUDE.md(数据规范)
写作场景 = Skills(Minimal) + MCP(None) + CLAUDE.md(写作助手)
```

切换场景，就像 iPhone 上切换 Focus Mode 一样简单。开始前端项目？一键应用「前端开发场景」。切到数据分析？再一键切换。Claude Code 的上下文中只会加载当前场景需要的工具，干净、高效。

## 五、使用场景段落（3个典型场景）

### 场景一：前端开发工作流

**背景**：我日常开发的项目以 React + TypeScript 为主，经常需要与设计师协作。

**我的配置**：

- **Skills**：TypeScript 语法辅助、React Hooks 最佳实践、CSS-in-JS 规范
- **MCP Servers**：
  - Figma MCP：直接在 Claude Code 中查看设计稿、获取设计规范
  - Vercel MCP：一键部署预览、查看部署状态
  - GitHub MCP：管理 Issue 和 PR
- **CLAUDE.md**：包含项目的代码规范、组件命名约定、Git 提交格式要求

**实际效果**：

当我打开一个前端项目时，Ensemble 会通过 Finder 右键菜单或项目关联功能，自动应用这套配置。Claude Code 获得了恰到好处的能力——能看设计稿、能部署、能管理代码，但不会被数据库工具、Python 脚本这些无关内容占用上下文。

### 场景二：数据分析工作流

**背景**：每周我都要做数据分析报告，涉及从数据库提取数据、用 Python 处理、生成可视化图表。

**我的配置**：

- **Skills**：Python 数据处理、SQL 查询优化、数据可视化指南
- **MCP Servers**：
  - PostgreSQL MCP：直接查询生产数据库
  - BigQuery MCP：处理大数据集
  - Notion MCP：将分析结果写入文档
- **CLAUDE.md**：数据处理规范、图表样式指南、敏感数据处理要求

**实际效果**：

分析数据时，Claude Code 能直接帮我写 SQL、优化查询、生成 Python 处理脚本，还能把结果整理到 Notion。而前端开发相关的工具完全不会出现，上下文空间全部留给真正需要的对话。

### 场景三：文档写作工作流

**背景**：写技术博客、产品文档时，我需要 Claude Code 专注于内容本身，而非代码工具。

**我的配置**：

- **Skills**：仅保留基础的文本处理能力
- **MCP Servers**：无（或仅保留 Notion MCP 用于发布）
- **CLAUDE.md**：写作助手指令，包含文风要求、格式规范、写作框架建议

**实际效果**：

这是一个极简场景。Claude Code 变成一个纯粹的写作伙伴，没有任何开发工具的干扰。上下文窗口几乎全部用于对话本身，可以进行更长、更深入的写作讨论。

## 六、技术实现段落（简要介绍 Tauri + Symlink）

### 技术选型：为什么是 Tauri

Ensemble 使用 Tauri 2.0 框架开发——这是一个值得效率控关注的技术选择。

**与 Electron 的对比**：

大多数跨平台桌面应用使用 Electron（VS Code、Slack、Discord 都是如此）。Electron 的问题在于它需要打包一个完整的 Chromium 浏览器，导致应用体积动辄 100MB+，内存占用也相当可观。

Tauri 走了一条不同的路：

- **前端**：依然使用 React、Vue 等现代框架（Ensemble 使用 React）
- **后端**：用 Rust 编写，直接编译成原生代码
- **渲染**：使用系统自带的 WebView（macOS 上是 WebKit）

结果是：Ensemble 的安装包只有约 10MB，运行时内存占用约为同类 Electron 应用的 1/5。对于一个需要常驻后台的配置管理工具来说，这种轻量化至关重要。

### Symlink：无侵入式配置管理

Ensemble 管理配置的核心机制是 **符号链接（Symlink）**。

传统的配置管理往往需要直接修改原始配置文件，这会带来几个问题：
1. 配置被改乱后难以恢复
2. 多套配置之间难以切换
3. 容易与其他工具产生冲突

Ensemble 的做法是：
1. 在 Ensemble 内部维护多套配置文件
2. 通过符号链接将当前激活的配置「链接」到 Claude Code 期望的位置
3. 切换场景时，只需更新符号链接的指向

这种方式的好处是：
- **非破坏性**：原始配置文件始终安全
- **即时生效**：切换场景几乎是瞬时完成
- **易于回滚**：随时可以恢复到任何之前的配置状态

技术上看起来简单，但这种简单正是好设计的体现——用最小的复杂度解决实际问题。

## 七、结语段落

### 开源的初心

Ensemble 完全开源，基于 MIT 许可证发布。

做这个决定的原因很简单：作为一个长期使用开源工具的开发者，我深知好的效率工具应该是公共财产。Claude Code 本身就建立在开放的 MCP 协议之上，Ensemble 作为它的配套工具，自然也应该回馈社区。

源代码托管在 GitHub 上，欢迎查阅、使用、提交 Issue 和 Pull Request。无论是发现 Bug、提出功能建议，还是贡献代码，我都非常期待与你交流。

### 当前的局限

诚实地说，Ensemble 目前还有一些局限：

- **仅支持 macOS**：由于开发资源有限，暂时只支持 macOS 平台。Windows 和 Linux 的支持在计划中，但需要时间。
- **早期阶段**：可能存在一些边缘情况的 Bug，配置迁移等功能还在完善中。
- **需要基础的 Claude Code 知识**：如果你完全没用过 Claude Code，可能需要先熟悉基础概念。

接下来我计划：
1. 完善场景分享功能——让用户可以导入/导出场景配置
2. 添加更多预设场景模板
3. 探索 Windows/Linux 支持的可能性
4. 根据社区反馈持续迭代
